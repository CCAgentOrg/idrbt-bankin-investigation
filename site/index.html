What happened

In one sentence: the registry meant to be the trust anchor for Indian banking domains exposed every one of its own employees’ credentials to the open internet for over a year.

The IDRBT Domain Registration Portal at registrar.idrbt.ac.in — the exclusive registry for .bank.in domains — exposed its entire REST API without authentication. Anyone with curl could retrieve bcrypt password hashes, mobile numbers, email addresses, login IPs, and device fingerprints for every bank employee registered in the system. The portal was built by IKCON Technologies without any public tender, in violation of IDRBT’s own procurement handbook.

The vulnerability was reported to CERT-In within 25 minutes of discovery (June 8, 2026). CERT-In confirmed the fix on June 25, 2026. The deeper systemic issues remain unaddressed.

Attack scenarios

What a malicious actor could have done with data that was already public.

1. Phishing at Scale

Time required: <1 hour

Download 5,461 records with names, emails, mobile numbers, and organization names. Craft personalised spear-phishing emails referencing correct internal details only the real portal should know.

2. Domain Hijacking

Time required: Days to weeks

1,072 orphan Super Admin accounts — with no organization association — had bcrypt hashes exposed. Weak passwords crackable in hours. With Super Admin access: transfer domains, change DNS, lock out legitimate admins.

3. The Invisible Redirect

Time required: Minutes with portal access

Change a cooperative bank’s .bank.in A record to a phishing server. Free Let’s Encrypt certificate + green padlock + .bank.in domain = the customer sees nothing wrong.

4. Insider Threat Amplification

Time required: Instant

Device fingerprints and login IPs map exactly which employees have Super Admin powers, which organizations have the weakest security, and which accounts are stale and ripe for takeover.

Key takeaway

None of these required a zero-day, a firewall bypass, or a single line of exploit code. The attacker only needed curl. The data was already public.

Systemic issues

The fix addresses API exposure. These deeper problems remain.
🏛️
No public tender. IKCON Technologies was appointed without any RFP, violating IDRBT’s own 2015 procurement handbook. IKCON held 22 accounts including 3 Super Admins.
🔄
Test/production overlap. Phantom domains (VKTEST, IKCONTESTBANK) and gibberish accounts sit alongside real banks in the production database. Several have live SSL certificates.
🌐
No security baseline. Unlike global .bank TLD, .bank.in mandates zero security controls: 80% of cooperative banks lack DNSSEC, 40% have no DMARC, 47% have no HSTS.
📡
Data residency violations. Cooperative banks host customer-facing websites on foreign servers (US, Singapore, Lithuania) — undermining RBI data localization rules.
📋
IDRBT’s own claims contradicted. The portal’s privacy policy claimed “protected zones with firewalls and IPS” and “audited for known vulnerabilities.” The unauthenticated API was present from day one.
📄
Academic research ignored. IDRBT’s own researchers published papers on domain security, DNSSEC, and cybersecurity benchmarks. None informed the .bank.in implementation.
