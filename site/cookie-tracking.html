.bank.in Supercookies: How India's Banking Domain Becomes a Cross-Bank Tracking Vector

CashlessConsumer  ·  June 30, 2026
← Back to main investigation

A domain suffix designed to be a trust anchor — telling citizens "this website is genuinely your bank" — can accidentally become a cross-bank tracking surface when the browser doesn't know that sbi.bank.in and hdfc.bank.in should be treated as separate sites.

This page documents the browser cookie mechanics, the Public Suffix List fix that already exists, the gap that remains in practice, what Kingsly J discovered in Feb 2026, and an expedition plan to find real-world abuses.

Status: .bank.in was added to the Public Suffix List in Nov 2025 (PR #2603 by NIC.IN). But this doesn't eliminate the risk — PSL updates propagate slowly to browsers, WebViews, and embedded browsers. Kingsly found cookies crossing bank boundaries as late as Feb 2026, 3+ months after the PSL entry.

1. The Public Suffix List & Cookie Isolation

The Public Suffix List (PSL) is a machine-readable catalogue that tells browsers which domain suffixes belong to a registry — so cookie.example.co.uk and other.example.co.uk can't read each other's cookies.

Without a PSL entry for bank.in, the browser treats it as a "private domain":

The mechanic is simple: any cookie set with Domain=.bank.in by one bank will be sent to every *.bank.in domain the user visits. This means:

  • A session cookie set by SBI's site can be read by HDFC's site
  • A tracking cookie (ad ID, analytics fingerprint) follows the user across every bank visit
  • A malicious bank (or a compromised third-party script on any bank's page) can set cookies readable by all other banks

2. .bank.in on the PSL — Already Fixed at the Registry Level

During our research for this page, we discovered that bank.in is already on the Public Suffix List.

It was added via PR #2603 — "add bank.in and fin.in to the public suffix list" — submitted by Gaurav Kansal (NIC.IN), the .in registry operator, on 2025-11-06 and merged the same day by PSL maintainer Simon Friedberger.

This was part of the broader update by NIXI/NIC.IN to bring all official .in second-level domains onto the PSL — covering bank.in, fin.in, ac.in, gov.in, edu.in, and others added in the same batch (PR #1588, then PR #2603).

Answer to who can submit: NIC.IN, as the .in registry operator, already handled this. RBI/IDRBT are not the right party — the PSL submission is the registry's responsibility. The PSL submission was done by NIC.IN (the registry for .in), not RBI. So no PR is needed from us — it's done.

3. The Gap That Remains — PSL Updates Are Not Instant

Adding bank.in to the PSL repository is necessary but not sufficient. Here's why the risk persists:

3a. Browser Release Cycles

The PSL is shipped with browser releases. Chromium fetches updates every ~4-6 weeks for Chrome; Firefox ships with each release (~4 weeks). Safari and mobile WebViews update on their own cadence. A user running a 3-month-old browser in Feb 2026 would not have the Nov 2025 PSL update.

3b. Embedded Browsers & WebViews

Many Indian apps — banking apps, UPI apps, fintech — use embedded WebViews to render bank pages. These WebViews often use a bundled Chromium WebView that may not receive PSL updates for months or years. An SBI YONO app WebView from early 2025 would never know about bank.in on the PSL.

3c. Third-Party Tracking Scripts

The most dangerous vector: if any .bank.in site loads a third-party script (analytics, ad network, fraud detection, support widget), that script can set Domain=.bank.in cookies. Since browsers without the PSL update see bank.in as a single "site," the third-party can correlate user behaviour across SBI, HDFC, ICICI, and all other banks.

4. Kingsly's Discovery — Feb 27, 2026

🧵 While browsing SBI in incognito, noticed cookies scoped to .bank.in appearing — specifically _iidt, _vid_t, and WZRK_S_.... The dot-prefix (.bank.in not sbi.bank.in) means these cookies are shared across ALL *.bank.in domains.
Set-Cookie: _iidt=abc123; Domain=.bank.in; Path=/; Max-Age=31536000
Set-Cookie: _vid_t=xyz789; Domain=.bank.in; Path=/; Max-Age=31536000
Set-Cookie: WZRK_S_...=...; Domain=.bank.in; Path=/; Max-Age=31536000
— @kingslyj, Feb 27, 2026

Kingsly's finding is significant because:

  • The cookies existed 3+ months after PSL entry — meaning either the user's browser hadn't picked up the PSL update, or the site was still setting cookies in a way that assumed no PSL protection
  • Cookie naming suggests a tracker: _iidt and _vid_t follow the pattern of visitor ID / device ID cookies used by ad-tech and analytics platforms
  • WZRK_S_... is particularly suspicious — this naming convention matches WizRocket (now part of CleverTap), a customer engagement and analytics platform widely used by Indian fintech companies

5. Theoretical Privacy Harms

If a tracker successfully sets a .bank.in-scoped cookie, the following cross-bank profiling becomes possible:

HarmHow It WorksSeverity
Cross-bank browsing profile A single visitor ID cookie links all your bank visits — SBI, HDFC, ICICI, Axis — into one profile High
Fintech credit inference Visiting a bank known for personal loans → targeting you with debt products. Visiting multiple banks → tracked as "rate shopping" High
Session hijacking surface If any bank's site has an XSS, the attacker can read cookies set by other banks — because they're all scoped to .bank.in Critical
Identity stitching Ad-tech platforms can correlate device fingerprints across banking sessions where the user is authenticated, linking demographic data to browsing behaviour High
Competitive intelligence One bank can observe when a user browses a competitor's loan or credit card page — enabling real-time counter-offers in the same session Medium
Fingerprinting amplification Cookies combined with canvas/WebGL fingerprinting from each bank page creates a richer, harder-to-evade cross-site fingerprint High
Important: These are theoretical harms at this stage. The point of the expedition below is to find evidence of actual abuse.

Any third-party script loaded on a .bank.in page can set Domain=.bank.in cookies via document.cookie if the browser doesn't have the updated PSL. This is not just a theoretical risk — it's a browser-level affordance that any tracker can exploit.

7. 🕵️ Expedition: Find Actual Cookie Abuse

Now we need to go beyond theory. Here's the methodology to find real-world evidence:

Objective

Detect cookies set with Domain=.bank.in (or no specific domain restriction) on *.bank.in sites that could enable cross-bank tracking.

Methodology

  1. Visit a set of .bank.in domains — SBI, HDFC, ICICI, Axis, Kotak — in a browser with developer tools open
  2. Inspect all cookies set during page load — especially those with Domain=.bank.in or no Domain attribute (which defaults to the host, but scripts can widen)
  3. Check third-party requests — look for analytics pixels, tracking scripts, marketing platforms loading on banking pages
  4. Check if third-party scripts use document.cookie to set wide-scope cookies
  5. Verify cross-bank leakage — clear bank-specific cookies, visit bank A, note cookies set, visit bank B, check if bank A's cookies appear in bank B's request headers

Cookie Names to Watch

  • WZRK_* — CleverTap/WizRocket (engagement platform, widely used in Indian fintech)
  • _iidt — device/visitor ID
  • _vid_t — visitor ID, timestamp-based
  • _ga, _gid — Google Analytics (common on bank websites)
  • _fbp — Facebook Pixel
  • ajs_* — Segment analytics
  • Any cookie with Domain=.bank.in that is not a session-identifier specific to that one bank
