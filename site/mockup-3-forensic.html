RBI created .bank.in as a trust anchor — a domain suffix that tells citizens, "this website is genuinely your bank." The portal that issues these banking domains was built without a public tender, without a published security baseline, and without any vulnerability disclosure program. For over a year, it leaked every credential it held.

I. Security Claims vs Reality

IDRBT's published Security Policy makes three specific claims. All three are verifiably false.

❌ Claim #1 — "Audited before launch"
The policy states the site "was audited for known application-level vulnerabilities before the launch, and all the known vulnerability was addressed." 33+ unauthenticated endpoints exposing the full user database, billing records, and system config is a failure no competent audit could miss. No vulnerability disclosure program existed.
❌ Claim #2 — "Content is authenticated"
Every discovered endpoint required zero authentication — no login, no API key, no session cookie, no token. One curl command was sufficient.
❌ Claim #3 — "Procedural safeguards"
Exposing bcrypt hashes and PII of 5,576 bank employees violates the Digital Personal Data Protection Act, 2023, and RBI's own cybersecurity framework (RBI/2023-24/90).
Bottom line: IDRBT's published security policy is a work of fiction. Three specific claims made to the public were verifiably false.
